最終更新: 2026-04-17
目的: 中高層ボリューム検討を「斜線制限だけ」から「斜線制限＋天空率緩和」まで踏み込めるツールに拡張する。UI は別途ベスト案を検討中のため、本書は **中身（エンジン・AI・データ・計算フロー）の情報整理** のみ。

## 1. ゴール

### 1.1 最終ゴール
**「敷地情報を入れたら、その敷地で建てられる最大ボリュームが即出る」**

- 斜線制限ベースの最大ボリューム（現状 volume-check が出せる）
- **天空率緩和を活用した最大ボリューム** ← 新規
- 両者を並置し、「あと +N㎡ / +N 階積める」と提示

### 1.2 なぜ天空率か
建築基準法 **56条7項** は、同条1項（道路・隣地・北側斜線）と同条1〜5項（日影規制）の制限を、**天空率が従来斜線制限と同等以上** であれば適用除外にできる。

つまり、中高層でボリュームを最大化するには「斜線で止まらず天空率まで踏み込む」必要があり、現在の volume-check は斜線の手前で止まっている。

### 1.3 ユーザーが受け取る価値
- **時間短縮**: 従来 手計算 + Excel + CAD で半日〜1日 → 数分
- **判断支援**: 「このプランで天空率クリアするか」を即判定
- **ボリューム最大化**: 斜線案 vs 天空率案の比較で、実際に建てられる最大値を把握

## 3. 天空率エンジン仕様（新規実装すべきもの）

### 3.1 法令・告示根拠

| 文書 | 内容 |
|---|---|
| 建築基準法 56条7項 | 天空率緩和の根拠条文 |
| 建築基準法施行令 135条の5〜9 | 天空率の計算方法 |
| 平成14年国交省告示 第1350号 | 算定方法の詳細 |
| JIS Z 8204 / 日本建築学会『建築物のボリューム計算の方法』| 実装の参照文献 |

### 3.2 定義

天空率 = **(水平面半球の中で、建築物で遮られていない部分の立体角) ÷ (半球全体の立体角)**

- 測定点（ポジション）から見上げた時の「空の見える割合」
- 0.0（全遮蔽）〜 1.0（完全開放）
- 建築物を配置した状態の天空率 ≧ 斜線制限いっぱいに建てた時の天空率 → 緩和 OK

### 3.3 測定点の取り方

| 制限 | 測定点の位置 | 本数 |
|---|---|---|
| 道路斜線 | 前面道路の反対側境界線上、道路中心線に垂直 | 道路長に応じ複数、一般に 1m〜4m 間隔 |
| 隣地斜線 | 隣地境界線から水平距離で 16m / 12.4m（用途による）外側 | 境界線上に沿って |
| 北側斜線 | 北側境界線から 4m / 8m 外側 | 境界線上に沿って |

いずれも高さは 1.5m（一般に）地面から。

### 3.4 立体角計算アルゴリズム（2案）

#### 案 A: 解析的（正確・推奨）
1. 建築物の全面を測定点座標系へ射影
2. 各面を単位球面上に投影（stereographic 投影か gnomonic）
3. 球面多角形の面積（立体角）を Girard's theorem で算出
4. 合計立体角 ÷ 2π = 遮蔽率 → 1 - 遮蔽率 = 天空率

#### 案 B: レイトレース（実装容易・近似）
1. 半球を N × M（例 180×360）のレイに離散化
2. 各レイが建築物に当たるか判定（GPU / WebGL でシェーダ化可能）
3. 当たらない比率 = 天空率

**推奨**: まず案 B で WebGL の `gl.readPixels` を使って高速実装 → 精度検証 → 必要なら案 A に切替。

### 3.5 判定ロジック

```ts
function judgeSkyFactor(
  building: BuildingMesh,
  site: SiteBoundary,
  road: Road,
): {
  skyFactor: number;        // 建築物を入れた時の天空率
  skyFactorLimit: number;   // 斜線制限限界時の天空率（基準値）
  pass: boolean;            // 緩和適用可か
  margin: number;           // 基準値との差
}
```

基準値 = 「その敷地で斜線制限通りに建てた仮想の最大 envelope」の天空率。
実建築物の天空率 ≧ 基準値 なら緩和 OK。

### 3.6 最大ボリューム探索（これが本命）

```ts
function optimizeMaxVolume(
  site: SiteBoundary,
  roads: Road[],
  zoning: ZoningData,
  options: { strategy: 'slant' | 'sky-factor'; maxIterations?: number },
): VolumeResult
```

**戦略**:
- `slant`: 現状のエンジン（envelope.ts）で斜線いっぱい
- `sky-factor`: 斜線 envelope を出発点に、各階の平面を少しずつ拡張して天空率がクリアする限界まで探索
  - **局所探索**: 階層ごとにセットバック距離を減らして試す
  - **方位ごと**: 道路側/隣地側/北側で独立に探索
  - **終了条件**: 天空率が基準値を下回った時点、または容積率/建ぺい率の上限

| Phase | 内容 | 工数見積 |

| 0-A | 測定点生成ユーティリティ | 2 日 |
| 0-B | 立体角計算（案 B: レイトレース） | 2 日 |

## 4. AI の使いどころ（新規 + 既存の強化）

### 4.1 既に AI が効いている
- **測量図 OCR → 敷地抽出**（Gemini Vision）: そのまま流用。PDF/画像投げれば頂点・道路・用途まで取れる
- **チャット Q&A**: 法規解釈・計画相談

### 4.2 AI を追加で使うべき箇所

| 用途 | AI モデル | 価値 |
|---|---|---|
| **「住所ください」→ 全部 AI で埋める** | Gemini（複数 API を裏で叩く） | 現在は各 API を手動選択。AI が「住所で geocode → PLATEAU → 用途地域取得 → 道路取得」を自動化 |
| **「こんな用途の建物建てたい」→ 最適プラン提案** | Claude / Gemini + エンジン | 「オフィスで 2000㎡ 欲しい」と言えば、天空率も使って提案を複数案出す |
| **計算書の自動コメント生成** | Claude | 「本案は北側斜線で 1.2m 追加緩和、結果 +1 階」等の説明文を自動生成 |
| **図面読み取り範囲拡張** | Gemini Vision | 現在は測量図のみ。配置図・確認申請書・地盤調査報告書も読めるように |
| **斜線/天空率の異議検討** | Claude（ロング思考） | 役所指導を想定して「この案は指摘される可能性がある」を事前警告 |

### 4.3 AI を使わなくていい箇所
- エンジン計算（envelope・天空率立体角）: 純幾何計算。決定論的に実装
- 3D 表示: Three.js
- PDF 出力: print スタイルで静的

## 7. 技術的論点

### 7.1 WebGL レイトレース実装
- Three.js の `WebGLRenderer.readRenderTargetPixels` で OK
- 1 測定点あたり 180×360 = 64,800 レイ、現代 GPU で 1ms 未満
- 30 測定点で 30ms、ユーザ待ち感ゼロ

### 7.2 天空率基準値（斜線限界 envelope）の天空率
- 既存 `envelope.ts` が斜線限界の 3D メッシュを出す
- これを測定点から見上げて立体角計算 → 基準値
- つまり同じレイトレース関数で 2 回流すだけ

### 7.3 最適化戦略
- 探索パラメータが多い（階高、コア位置、セットバック距離×方位）
- ナイーブに線形探索すると組合せ爆発
- **推奨**: 階ごと・方位ごとに独立最適化 → 局所最適でも実用十分
- 将来的に遺伝的アルゴリズムやベイズ最適化で大域最適

### 7.4 計算負荷の対策
- Web Worker でバックグラウンド実行
- プログレス UI（「探索中 3/8 方位」）
- キャッシュ（同一入力 → 同一結果）

### 7.5 精度検証
- 市販ソフト（TP-PLANNER、ADS-win、ふぃっと）との突合
- 日本建築士会連合会の標準問題集での突合
- 役所指導案件 3 件以上で実測突合
